# AGENTS.md

The free core: the repository / build-cache server modules consumed by downstream editions. Design rationale lives in [`DESIGN.md`](DESIGN.md).

## Build & test

- `java build/jenesis/Project.java` resolves, compiles, and tests every module. **Requires JDK 25** (the sources use module-import declarations and unnamed variables).
- Fast iteration — build one module's subgraph with a `+` selector: `+source+<path>` or `+test+<path>` (e.g. `+test+store+s3`; a nested `store/s3` is written `+store+s3`, and `/<step>` drills into a step). `+` selectors are *lenient* — a wrong one silently matches nothing, so confirm your module shows an `[EXECUTED]` line.
- `-Djenesis.*` system properties go **before** the `Project.java` path; bare selectors go **after** it.
- `java build/jenesis/Project.java help` documents every selector and `-D` flag (`pin` regenerates the `@jenesis.pin` version/checksum lines; `export` produces the deliverable repository).

## Local gotchas (a red here is often the environment, not a regression)

- Container-backed tests (MinIO / Nexus, driven through the `docker` CLI) **self-skip without Docker**; the strict CI lane (`-Djenesis.project.properties=ci`) makes them *fail* instead.
- A few tests need a **UTF-8 locale** (non-ASCII path cases) or a **`GITHUB_TOKEN`** (the live advisory-feed smoke). Set `LANG=C.UTF-8` for the former.

## Engineering principles (operative summary)

The full pass/fail checklist is [`PRINCIPLES.md`](PRINCIPLES.md) — run it by eye over every diff. The durable contract in brief (this free core is the shared base downstream editions reuse, so keep the seams clean and never fork mechanism downstream):

- **Stream, never buffer (§1).** No whole-blob read (`readAllBytes`/`toByteArray`) of an *artifact* on any upload/download/proxy path — only a small index/metadata blob may be fully materialised. A hosted publish is store-then-gate (hash-on-write into CAS, gate a descriptor, not a `byte[]`); proxy uses the streaming `download` path; `HEAD` answers from metadata; heap stays bounded under a >2 GiB upload.
- **Thin core, pluggable SPI (§2).** A `format/*` / `compliance/*` module requires only its SPI, never a peer plugin or the server; the server `uses` SPIs and `requires` no concrete plugin. New capabilities are `ServiceLoader`-discovered and SPI-named — not a core edit. No focal class enumerates plugged concepts by hand (formats, inspectors, feeds, hold kinds, settings, screens fan out through their SPI). SPI *contract* modules stay `java.base`-light; the heavy deps ride the impl/bundle. No new database — all durable state is `ArtifactStore` objects.
- **Persist only through the store; converge idempotently (§4).** All persistence goes through the store SPI (`ArtifactStore` / `CacheStorage`) — no raw `File`/`Path`, no direct cloud SDK outside a provider module. Everything durable is a store object: CAS for immutable content, `writeVersioned` (compare-and-set, check the result and retry on conflict) for mutable pointers, a `Lease` for single-writer sweeps. Every derived surface either converges after a crash through a `Lease`-guarded idempotent repair pass, or its javadoc states it is best-effort and why.
- **Self-healing — converge even when added late (§5).** A capability switched on *late*, over a store that already holds artifacts from before the plugin existed, back-fills its derived state from the durable store on first activation (same `Lease`-guarded pass) and re-checks on each start — adopting a plugin is a config flip, never a re-import or manual migration. Where it genuinely cannot back-fill it degrades gracefully and says so through its `NONE`/empty sentinel or health surface; it **never serves a silently-incomplete view as if it were whole**. Open-ended kinds (formats, feeds, hold kinds) survive a plugin added at any point in the product's life.
- **Read-first performance (§7).** The reader pays for nothing a writer or sweep could pre-do; precompute/cache/roll-up on write or in a background sweep. Prefix listing and direct key lookups, never full scans. Generated-on-read output (POM, `maven-metadata.xml`, indexes) is cacheable and ETag-revalidatable.
- **Prefer libraries over hand-rolling (§8).** Reach for a maintained, permissively-licensed library for parsers/codecs/compression/crypto/hashing/encoding/HTTP first. Hand-roll only when nothing fits the module path / native-image / licence — then keep it minimal, say why in the code, and pin it to the reference output with a test.
- **Fail fast; make errors visible (§9).** A capability an operator *explicitly selected* that cannot be honoured (module absent, name misspelled, required config unset) throws at resolution/startup — never a silent fallback (the store is the exemplar: `store=s3` with the s3 module off must not boot against the local filesystem). A config error names what was selected and what is missing. No swallowed failure on a correctness-bearing path: an exception whose loss would change what is served, held or counted is logged/countered/surfaced, never caught into a silent default.
- **Reads render, writes refresh (§10).** A GET renders only what is durably stored — no external fetch/scan/write on the read path, and the view stands when the source is down. Refreshing feeds/upstreams/scanners is an explicit, idempotent, write-role action (button or `refresh=true`), beside the scheduled sweeps. Every derived/externally-sourced view shows its last fetch/scan instant so an empty panel is never ambiguous between "clean" and "never scanned". A caller lacking the write role sees no refresh control but still sees the staleness.
- **No backwards-compatibility shims.** Defaulting-secure and clean cutovers are preferred over compat layers; there is no requirement to preserve an old wire/format/behaviour across a change.
- **Flag, don't force.** If a feature can only be met by fighting these gates, mark it flagged for a human decision rather than hand-rolling a workaround that fights them.
